\# 11. ТЕХНОЛОГИЧЕСКИЙ СТЕК (TECH STACK CORE)



\## 11.1. Frontend: Telegram Mini App (TMA)

\* \*\*L1 (Информация)\*\*: Визуальная часть (клиент) пишется на \*\*React.js + TypeScript\*\* (сборщик Vite).

\* \*\*L2 (Пояснение)\*\*: Это обеспечивает мгновенную загрузку в WebView мессенджера. Обязательно использование библиотеки `@twa-dev/sdk` (или официального скрипта Telegram WebApp) для инициализации `initData` и получения системных цветов. Стилизация выполняется через \*\*Tailwind CSS\*\* для максимальной скорости верстки и поддержки Dark/Light mode Telegram. Использование тяжелых фреймворков (Flutter Web, Angular) строго запрещено.



\## 11.2. Backend: Serverless Logic

\* \*\*L1 (Информация)\*\*: Серверная бизнес-логика реализуется через \*\*Firebase Cloud Functions (Node.js + TypeScript)\*\*.

\* \*\*L2 (Пояснение)\*\*: Использование единого языка (TypeScript) на клиенте и сервере исключает ошибки типизации. Бэкенд отвечает за критические узлы: скрипт-оракул Fragment (запуск по cron каждые 4 часа), расчет сплита 20% (запись транзакций в Firestore), и валидацию `initData` от Telegram для защиты от подделки сессий.



\## 11.3. Database \& Real-time Sync

\* \*\*L1 (Информация)\*\*: База данных — \*\*Firebase Firestore\*\* (NoSQL).

\* \*\*L2 (Пояснение)\*\*: Firestore выбран за встроенную поддержку WebSockets. Это критически важно для механики "60-sec Undo" (отмена сделки) и экрана "Радар Данбара", где данные должны обновляться на экранах пользователей мгновенно в реальном времени (Real-time listeners), без ручного обновления страницы.



\## 11.4. Hosting \& Deployment

\* \*\*L1 (Информация)\*\*: Фронтенд размещается на \*\*Firebase Hosting\*\*.

\* \*\*L2 (Пояснение)\*\*: Это дает бесплатный SSL-сертификат (обязательное требование Telegram для Mini Apps), глобальную CDN для быстрой загрузки в любой точке мира (от Дубая до Вьетнама) и бесшовную интеграцию с Cloud Functions и Firestore.
